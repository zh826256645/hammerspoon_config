# Hammerspoon 配置治理方案

## 1. 目标

本方案用于让配置保持“看得懂、改得动、能验证”，不追求把个人配置改造成通用框架。

治理完成后应满足：

- `init.lua` 能准确说明当前配置启用了什么，运行状态能通过 Console 或功能提示判断。
- README 与实际行为一致，并明确外部依赖。
- 本机相关参数有固定位置，不再散落到多个功能模块。
- 模块之间的依赖可追踪，不依赖隐含的加载顺序猜测。
- 每次修改都有与风险匹配的最小验证方法。

## 2. 治理基线问题

制定本方案时，目录结构已经按功能拆分，入口也集中在 `init.lua`，基础结构并不混乱。主要问题来自长期增量修改：

1. **文档漂移**：README 描述了当前未加载的粘贴板历史，并保留了旧的 Wi-Fi 与代理配置名称。
2. **启用状态不清晰**：模块文件存在不代表功能已启用，必须同时检查 `init.lua` 中的 `require`、注册和 `start` 调用。
3. **机器配置分散**：天气路径、Python 路径、蓝牙设备、SSID、代理配置等本机参数分布在不同模块中。
4. **全局依赖较多**：快捷键、应用 ID、控制函数和 watcher 实例通过全局变量共享，模块行为依赖加载顺序。
5. **验证入口不足**：当前没有统一的语法检查或运行检查说明，修改后主要依赖人工 Reload Config。

制定方案时已确认的证据：

| 位置 | 现状 | 影响 |
| --- | --- | --- |
| `init.lua` | 加载了帮助模块但未注册帮助菜单，未加载粘贴板历史 | 文件存在与功能启用不是同一件事 |
| `README.md` | 仍把粘贴板历史写为现有功能，Wi-Fi/代理名称与代码不一致 | 新读者会误判实际行为 |
| `modules/weather/weather.lua` | 包含城市、Python 和本地脚本路径 | 更换环境需要修改功能实现 |
| `modules/bluetooth/blueutils.lua`、`modules/wifi/wifi.lua` | 包含设备、SSID、代理和音量等本机参数 | 本机配置分散，修改容易漏项 |
| `modules/monitor/monitor.lua` | 直接使用其他模块提供的全局函数和常量 | 模块依赖加载顺序且难以单独验证 |

## 3. 治理原则

### 3.1 `init.lua` 是“配置启用”的唯一事实来源

- 只有被 `init.lua` 加载并启动的功能才算“配置启用”。
- “运行可用”还要求依赖和配置有效，必须由 Hammerspoon Console、通知或功能状态表达。
- README 功能清单默认只列配置启用的功能；因迁移等原因仍需提及的未启用功能必须显式标记。
- 不增加第二份模块开关清单，避免双重维护。

### 3.2 配置与行为分离，但只分离本机变量

需要集中管理的内容：

- 天气城市和本地脚本路径。
- 蓝牙设备标识。
- Wi-Fi SSID、代理配置名和家庭音量。

应用 Bundle ID、快捷键定义等稳定的功能常量继续留在所属模块，不为了形式统一全部搬走。

当前仓库公开：版本库只提交 `config.example.lua`，实际本机值放入被 `.gitignore` 忽略的 `config.lua`。配置文件不得保存密码、令牌或其他秘密。

### 3.3 模块边界按真实职责收口

- 新增或治理过的模块应返回一个 Lua table，公开最少的注册或控制函数。
- 模块内部状态默认使用 `local`。
- 跨模块调用必须能从 `require` 和调用点追踪，不新增 interface、factory 或兼容 wrapper。
- 每次只原子治理一个模块及其直接调用方，不做一次性全量改写。
- `require` 只加载定义，不启动 watcher、timer、hotkey 或 menubar；启动由 `register`/`start` 明确触发。
- 能启动持续任务的模块应提供对应 `stop`，负责停止 watcher/timer 并取消未执行的延迟任务。
- 集中配置或切断一条跨模块依赖时可以同时修改多个直接相关文件，但必须作为一个可独立验证的增量完成。

### 3.4 删除优先于保留“以后可能用”的代码

- 未启用功能只有 README 或工单中记录了用途、启用入口和验证方法时才保留。
- 注释掉的旧应用快捷键、未使用导入和废弃辅助函数应在相关模块被修改时清理。
- 不建立 `disabled/` 长期存放废弃实现；Git 历史已经承担归档职责。

### 3.5 验证与风险匹配

- 文档变更：核对 README、`init.lua` 和模块实现的一致性。
- 纯配置变更：Reload Config 后验证对应功能并检查 Hammerspoon Console。
- 分支、定时、状态逻辑变更：保留一个能复现关键行为的最小断言或检查。
- 涉及睡眠、网络、蓝牙的变更：至少验证一次取消路径，避免过期 timer 在唤醒后继续执行。

## 4. 分阶段治理

### 阶段 1：消除事实不一致（优先级 P0）

行动：

- 以 `init.lua` 为准重写 README 的“当前启用功能”。
- 明确帮助菜单和粘贴板历史当前未启用。
- 更新 Wi-Fi、代理配置和工作/娱乐模式说明。
- 补充天气、Python、`blueutil`、Sparkle/Mihomo 等外部依赖。

验收：新读者只看 README 和 `init.lua`，能正确回答“启用了什么、依赖什么、如何关闭”。

### 阶段 2：清理失效路径（优先级 P0）

行动：

- 确认粘贴板历史和帮助菜单是否仍有保留价值；若 README 或工单中没有用途、启用入口和验证方法则删除。
- 删除已替代的注释代码、未使用常量和未使用模块导入。
- 修正明显的命名错误时同步修改全部调用方，不保留旧别名。

验收：搜索未启用模块时都能找到对应的用途、启用入口和验证方法；否则模块已删除。仓库内不存在仅为兼容旧名字保留的透传代码。

### 阶段 3：集中本机配置（优先级 P1）

行动：

- 提交最小模板 `config.example.lua`，本机 `config.lua` 只返回配置 table 且不入库。
- 迁移天气城市、Python/脚本路径、蓝牙设备、SSID、代理配置和家庭音量等本机变量。
- 配置字段由使用它的功能校验；字段缺失时停止该功能并给出字段名，不阻止无关模块加载。
- 只有 `init.lua` 自身无法继续执行的入口错误才允许全局 fail-fast。
- 不把密码、令牌或其他秘密写入仓库。

验收：更换电脑或配置名称时，只需修改一个配置入口；缺少必要配置时错误信息能指出具体字段。

### 阶段 4：逐模块收敛全局依赖（优先级 P1）

推荐顺序：

1. `wifi`、`bluetooth`：先返回控制函数。
2. `application`：返回应用控制与快捷键注册函数。
3. `monitor`：显式引用上述模块，不再读取其全局函数和常量。
4. `weather`、`pasteboard`、`input_method`：把 watcher、menubar 和状态留在模块内部。

验收：被治理模块只暴露调用方实际使用的函数；删除对应全局变量后 Reload Config 行为不变。

### 阶段 5：建立最小验证基线（优先级 P1）

行动：

- 在 `luac` 与 Hammerspoon Lua 版本兼容时执行语法检查：`rg --files -g '*.lua' -0 | xargs -0 luac -p`。
- `luac` 不可用或版本不兼容时，记录 `verification_not_run_reason`，以 Reload Config 和 Console 作为替代检查。
- Reload Config 后检查 Console 无加载错误。
- 按改动范围执行下方 smoke checklist。
- 暂不引入测试框架或 CI；只有多人维护、频繁回归或自动发布出现时再增加。

验收：语法检查已通过，或已记录不能执行的原因与替代检查；Reload Config 后 Console 无新增加载错误；受影响的 smoke 项有明确结果。

Smoke checklist：

以下预期来自当前实现，治理时默认保持不变；如果需求主动改变，应同步更新实现、README 和对应 smoke 项。

- 窗口模块变更：吸附、最大化和跨屏移动各执行一次，窗口应落在目标位置且 Console 无错误。
- 模式或应用模块变更：手动切换工作/娱乐模式，VS Code 和 Codex 快捷键应只在工作模式可用。
- Wi-Fi 模块变更：连接公司/家庭 SSID，分别观察静音/配置声明的家庭音量及代理通知；10 秒内重复事件不应重复切换。
- 休眠模块变更：睡眠后 15 秒内唤醒，Console 应出现取消或跳过过期任务的信息，应用和网络不应被延迟关闭。
- 剪贴板模块变更：复制 `  test  `，剪贴板结果应为 `test`，且只显示一次复制提示。
- 天气模块变更：菜单能更新；景观脚本缺失时仅天气图生成失败，其他模块仍能加载。

验证结果记录在对应工单、提交说明或 PR 描述中；失败时停止当前改动，不继续叠加补丁。

## 5. 实施顺序与停止条件

| 顺序 | 工作项 | 价值 | 风险 |
| --- | --- | --- | --- |
| 1 | README 与启用状态对齐 | 立即消除误判 | 低 |
| 2 | 删除确认废弃的功能和注释路径 | 降低维护面 | 低 |
| 3 | 集中本机配置 | 降低迁移和修改成本 | 中 |
| 4 | 收敛 `monitor` 相关全局依赖 | 降低加载顺序耦合 | 中 |
| 5 | 按需治理其余模块 | 改善局部可维护性 | 中 |

满足以下任一条件时应停止当前治理项或停止继续扩大范围，并重新确认方案：

- 下一步只是为了目录或代码形式更统一，没有实际维护收益。
- 需要一次性重写多个稳定模块才能完成。
- 无法给出 Reload Config 后的具体行为验证方法。
- 当前治理项可以通过删除或更新文档解决，不应继续扩展为代码重构。

## 6. 日常变更清单

修改前：

- 确认功能是否在 `init.lua` 启用。
- 搜索直接调用方和共享全局变量。
- 明确本次要保持不变的行为。

修改后：

- 更新与代码事实直接相关的 README 内容。
- 执行最小语法或行为验证，并检查 Console。
- 检查最终 diff，不夹带无关格式化和重命名。
- 新增本机依赖时说明安装方式、失败表现和关闭方式。

## 7. 完成定义

单项治理工作只有同时满足以下条件才算完成：

- 目标问题已消失，而不是增加兼容层隐藏问题。
- `init.lua`、实现和 README 对同一功能的描述一致。
- 已执行与风险匹配的验证，并记录未验证项。
- 没有新增无调用方抽象、额外依赖或未来占位代码。
- Hammerspoon 其他已启用模块仍能正常加载。
